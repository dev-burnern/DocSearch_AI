# 관리자 운영 상태 화면 설계

## 목표

관리자가 DocSearch AI의 운영 준비 상태, 외부 의존성 점검 결과, 핵심 런타임 설정을 프론트엔드에서 한눈에 확인할 수 있게 한다. 포트폴리오 관점에서는 “동작하는 기능”뿐 아니라 “운영 가능한 서비스”라는 인상을 강화한다.

## 사용자 흐름

1. 사용자가 공통 API Key를 입력한다.
2. `/v1/workspace`로 워크스페이스와 역할을 확인한다.
3. 역할이 `admin`이면 감사 로그 탭과 함께 운영 상태 탭을 노출한다.
4. 관리자는 운영 상태 탭에서 새로고침 버튼을 눌러 현재 상태를 조회한다.
5. API는 readiness 결과와 안전한 설정 요약만 반환한다.

## 백엔드 설계

새 엔드포인트는 `GET /v1/admin/operations`로 둔다. 기존 `require_admin_workspace_context`를 그대로 사용해 관리자 역할만 접근할 수 있게 한다.

응답에는 다음 내용을 포함한다.

- `status`: 전체 준비 상태
- `service`: 서비스 이름
- `workspace`: 요청한 관리자 워크스페이스 정보
- `checks`: `/ready`에서 사용하는 설정/의존성 점검 목록
- `settings`: 노출해도 안전한 운영 설정 요약

민감 정보인 API Key, DB URL, MinIO Secret, LLM API Key는 반환하지 않는다.

## 프론트엔드 설계

새 API 클라이언트 `operations-api.ts`를 만들고, `OperationsStatusWorkspace` 컴포넌트를 추가한다. 컴포넌트는 기존 감사 로그 화면처럼 공통 API Key가 있으면 내부 입력을 숨기고, 없으면 자체 API Key 입력으로도 조회할 수 있게 한다.

화면 구성은 다음과 같다.

- 좌측: 조회 패널과 새로고침 버튼
- 우측: 전체 상태 요약, 핵심 설정 태그, 점검 결과 목록

기존 레이아웃 패턴에 맞춰 `Card`, `Tag`, `List`, `Alert`, `Empty`를 사용한다.

## 에러 처리

- API Key가 없으면 조회 버튼을 비활성화한다.
- 일반 사용자가 접근하면 백엔드가 403을 반환한다.
- 프론트엔드는 API 오류 메시지를 Alert로 표시한다.
- 상태가 `not_ready`여도 API 요청 자체가 성공하면 화면에 점검 결과를 표시한다.

## 테스트 전략

- 백엔드: admin 접근 성공, member 접근 거부, 설정 요약이 민감 정보를 제외하고 포함되는지 테스트한다.
- 프론트 API: 요청 경로, `X-API-Key` 헤더, 오류 메시지 파싱을 테스트한다.
- 프론트 UI: admin 역할에서 운영 상태 탭 노출, member 역할에서 숨김, 조회 성공/실패 화면을 테스트한다.
